Single-window Electron app split into three layers:
- `main.js` (Node/Electron main process): owns the BrowserWindow, registers the single-instance lock, persists `messages.json`/`settings.json` under `app.getPath('userData')`, copies uploaded files into `files/` (UUID-named) and voice recordings into `voice/`, serves them back via a custom `local-file://` protocol handler so the renderer never sees the filesystem.
- `preload.js`: bridges exactly one object (`window.messenger`) to the sandboxed renderer through `contextBridge`, exposing only IPC handles for store/settings/file pick/save/open/reveal/voice/notification/theme/fileUrl — no direct Node access.
- `renderer.js` + `index.html` + `styles.css`: pure browser-side UI. `renderer.js` holds all state in-memory (`state.messages[]`, `settings`), renders bubbles/day-dividers/reactions/pinned bar/search highlights, and mutates state then calls `api.save()`; `app.js` is a duplicate legacy renderer kept alongside it.

IPC channel naming follows a `<domain>:<action>` convention (`store:load`, `file:pick`, `voice:save`, etc.). Data schema is flat `{ messages: [{ id, text, files[], createdAt, read, reactions{}, pinned, edited }] }`; file metadata is `{ name, storedName, size, mime, category }`. Build/deploy via `electron-builder` targeting Windows NSIS + portable, with `assets/buildResources` as the icon directory.
